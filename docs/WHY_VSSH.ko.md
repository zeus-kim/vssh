# 왜 vssh인가 — 플릿과 AI 에이전트를 위한 SSH 대안

`vssh`는 "명령어가 짧아진 SSH"가 **아닙니다**. 다른 제품 추상화입니다: 사설망용
타입드·키 전용 원격 실행 데몬으로, 조작 주체가 점점 사람이 아니라 **AI
에이전트·자동화 런타임**이 되는 상황을 위해 설계됐습니다.

이미 `sshd`가 도는 박스에 대화형 셸만 열면 된다면 OpenSSH를 쓰세요 — `vssh`는
의도적으로 그것을 감싸지 않습니다. `vssh`는 그 주변의 모든 것을 위해 존재합니다:
타입드 액션 입력, 구조화된 증거 출력, 그리고 인증·정책·감사·장기 잡·터널·전체 플릿
상태를 1급 개념으로.

## 애초에 왜 SSH 대안인가?

SSH는 1995년에 사람이 한 머신에 타이핑하는 용도로 설계됐습니다. 현대의 플릿·
에이전트 자동화는 그 설계를 다섯 가지 방향에서 압박합니다:

1. **sshd가 필요 없어야 한다.** 모든 노드에 `sshd` + 사용자 + 호스트키 + PAM을
   세우는 건 크고 상태 많은 표면입니다. `vssh`는 정적 바이너리 하나와 사설망
   도달성만 필요합니다.
2. **출력은 텍스트 스트림이 아니라 데이터여야 한다.** LLM(또는 어떤 프로그램이든)이
   터미널 스트림을 파싱하는 건 추측입니다. `vssh`는
   `stdout/stderr/exit/duration/transport`를 타입드 봉투로 돌려줍니다.
3. **인가는 "셸 = root"보다 세밀해야 한다.** SSH 접근은 보통 전부-아니면-전무입니다.
   `vssh`는 키별 capability, 명령 allow/deny, 경로 스코프, 레이트 제한이 있습니다.
4. **액션은 구성상 감사 가능해야 한다.** `vssh`는 인증된 키에 귀속해 액션마다
   해시체인 기록을 남깁니다.
5. **장기·멀티노드 작업이 1급이어야 한다.** tmux/nohup 관습이 아니라. `vssh`는 잡,
   아티팩트, fan-out, capability 라우팅이 있습니다.

## ssh vs vssh

| | OpenSSH | vssh |
|---|---|---|
| **타깃 전제조건** | `sshd` + 사용자 + 호스트키 + PAM | 단일 정적 바이너리(`vssh server`), sshd 불필요 |
| **인증** | PAM 기반 비밀번호/키 | 노드별 Ed25519 챌린지–응답(VAUTH1); 공유 시크릿 없음 |
| **전송** | SSH 와이어 프로토콜 | TLS 1.3 + Ed25519 **raw-key 핀**(CA 없음) |
| **호스트 신원** | `known_hosts` TOFU | 핀된 name→key 레지스트리, **기본 ON**, 오라우팅 차단 |
| **명령 결과** | 원시 텍스트 스트림 | 타입 증거: stdout/stderr/exit/duration/transport |
| **인가** | 셸 = 전체 접근 | 키별 **capability** + 명령 **정책**(allow/deny, 경로, 레이트) |
| **감사** | 내장 없음 | 액션별 **해시체인** 기록 |
| **장기 작업** | tmux / nohup | `job_start/status/logs/cancel` + 아티팩트 수집 |
| **멀티노드** | `ssh` 스크립팅 | 네이티브 fan-out + capability/health **라우팅** |
| **주소지정** | `user@ip` | 노드 **이름** + capability/tag/health |
| **AI / 자동화** | 터미널 텍스트 파싱 | **MCP 네이티브** 타입 도구, 증거 봉투 |
| **플릿 상태** | 없음 | **서명·타임스탬프** 플릿 스냅샷, 노드 복제 가능 |
| **온보딩** | 호스트별 수동 설정 | 제로터치 자동 셋업; 한 명령 MCP 부착 |
| **터널** | `-L/-R/-D` | 동일한 인증·감사 채널 위의 `-L/-R/-D` |
| **키 회전** | 수동 | `vssh keygen --rotate` + 플릿 `authorized_keys` 도구 |

## 오늘 실제로 shipped된 것 (피치가 아니라 증거)

위 표의 모든 행이 실제 기능에 매핑됩니다:

- **키 전용 인증**: TLS 1.3 + Ed25519 VAUTH1; 데몬은 VAUTH1이 아닌 라인을 거부.
  코드 어디에도 공유 시크릿/HMAC 없음.
- **호스트 신원검증**: 기본 ON 핀 레지스트리로 이름이 엉뚱한 데몬으로 가는 걸 차단.
- **정책 + 감사**: deny-first 명령 분류, 키별 caps, 경로 스코프, 레이트 제한;
  변조 탐지 가능한 해시체인 감사 로그.
- **타입드 API**: `facts`, disk/mem/gpu/load/processes/logs/service/docker, 그리고
  통합 `node_info`(버전, uptime, 아치, 강제 적용 상태).
- **잡 + 아티팩트 + 터널(-L/-R/-D, MCP `vssh_tunnel` 도구 포함).**
- **플릿 상태**: 컨트롤러 서명·타임스탬프 스냅샷(인벤토리 + 노드 키 + caps +
  liveness), 어디서나 읽기·검증 가능, 노드 복제 가능.
- **제로터치 온보딩**: 최초 사용 시 셋업 자동 실행; `vssh mcp-install`이 한 명령으로
  AI 클라이언트에 vssh 부착.
- **AI 주도 설정(게이트)**: 명시적 opt-in 하에 에이전트가 로컬 설정 관리(키 인가/
  회수, 노드/IP 설정, 호스트키 핀).
- **키 회전 + 복구 런북**(`docs/KEY_ROTATION.md`).
- **배포**: 체크섬 검증 원라인 설치, `pip install vssh`, 리눅스 7아치 + macOS 릴리스
  (FreeBSD 실험적).

## OpenSSH가 여전히 맞는 경우

- 이미 `sshd`가 도는 박스에 사람이 대화형 터미널을 여는 경우.
- 알려진 호스트로의 즉석 셸 작업·포트포워드·파일 복사.
- 두 번째 데몬이나 타입드 컨트롤 플레인을 원치 않는 경우.

`vssh`는 거기서 경쟁하면 안 됩니다 — `ssh user@ip`를 감싸는 건 가치가 부족합니다.

## 제품 원칙

모든 `vssh` 기능은 최소 한 가지를 통과해야 합니다: sshd 없이 동작한다; 에이전트가
신뢰할 구조화된 증거를 만든다; SSH가 못 주는 정책/통제를 더한다; 장기·멀티노드
자동화를 개선한다; 노드/플릿 상태를 터미널 텍스트가 아니라 타입드 데이터로 노출한다.
기능이 그저 "SSH인데 더 짧음"이면 제품에 속하지 않습니다.
